# System Boundary Interface Documentation

This document describes the system's external invocation interfaces, including CLI commands, API endpoints, configuration parameters, and other boundary mechanisms.

## Command Line Interface (CLI)

### cortex-mem-cli vector prune

**Description**: Prune outdated or low-score vectors from memory collections to optimize storage and search performance.

**Source File**: `context-engine/binaries.ts`

**Options**:

- `--config`(string): required - Path to config.toml file
- `--tenant`(string): required - Tenant ID for multi-tenant isolation

**Usage Examples**:

```bash
cortex-mem-cli vector prune --config /path/to/config.toml --tenant tenant_claw
```

### cortex-mem-cli vector reindex

**Description**: Rebuild vector indexes to improve search accuracy and performance after schema or embedding model changes.

**Source File**: `context-engine/binaries.ts`

**Options**:

- `--config`(string): required - Path to config.toml file
- `--tenant`(string): required - Tenant ID for multi-tenant isolation

**Usage Examples**:

```bash
cortex-mem-cli vector reindex --config /path/to/config.toml --tenant tenant_claw
```

### cortex-mem-cli layers ensure-all

**Description**: Ensure all L0/L1/L2 memory layers are generated and synchronized for the specified tenant.

**Source File**: `context-engine/binaries.ts`

**Options**:

- `--config`(string): required - Path to config.toml file
- `--tenant`(string): required - Tenant ID for multi-tenant isolation

**Usage Examples**:

```bash
cortex-mem-cli layers ensure-all --config /path/to/config.toml --tenant tenant_claw
```

## API Interfaces

### POST http://localhost:8085

**Description**: cortex-mem-service HTTP API for semantic memory operations including ingestion, recall, and layer management. Configured via serviceUrl in config.toml.

**Source File**: `context-engine/client.ts, plugin/src/client.ts`

**Request Format**: JSON

**Response Format**: JSON

**Authentication**: API key from llm.api_key or embedding.api_key in config.toml

### POST http://localhost:6334

**Description**: Qdrant vector database API for storing and retrieving semantic embeddings. Accessed internally via CortexMemClient.

**Source File**: `plugin/src/client.ts, context-engine/client.ts`

**Request Format**: JSON

**Response Format**: JSON

**Authentication**: None (local service, assumed trusted network)

## Integration Suggestions

### Configuration Management

Use plugin-provided config values (e.g., llmApiKey, serviceUrl) to auto-update config.toml during plugin installation to reduce manual setup.

**Example Code**:

```
updateConfigFromPlugin({ llmApiKey: 'sk-...', embeddingApiBaseUrl: 'https://api.openai.com/v1' });
```

**Best Practices**:

- Validate required fields (api_key) before auto-updating
- Log config changes for auditability
- Provide fallback to defaults if plugin config is malformed

### Plugin Lifecycle

Leverage OpenClaw's registerHook('after_install') to auto-configure memory slot and disable built-in memory, ensuring seamless plugin adoption.

**Example Code**:

```
api.registerHook('after_install', async () => { await api.updateConfig({ 'plugins.slots.memory': 'none' }); });
```

**Best Practices**:

- Always check if updateConfig API exists before calling
- Use descriptive hook names to avoid conflicts
- Notify users via logger if auto-configuration fails

### Service Orchestration

Automatically start Qdrant and cortex-mem-service binaries on plugin load if autoStartServices=true, improving user experience.

**Example Code**:

```
if (config.autoStartServices) { await ensureAllServices(log); }
```

**Best Practices**:

- Check binary availability before attempting to start
- Gracefully degrade if services fail to start
- Log detailed error messages for troubleshooting
