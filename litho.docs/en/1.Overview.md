## 1. Project Introduction

### 1.1 Project Name and Description
**Project Name:** MemClaw  
**Description:** MemClaw is a modular plugin-based memory and context management system designed for integration within the OpenClaw ecosystem. It serves as the intelligence layer for AI agents, providing persistent memory, semantic search capabilities, and context retrieval. The system operates by orchestrating backend infrastructure (vector databases and memory services) while exposing a unified API to the host environment.

### 1.2 Core Functionality and Value
The primary value proposition of MemClaw lies in enabling stateful, long-term memory for autonomous agents. Key functional capabilities include:
*   **Semantic Search:** Tiered indexing (L0/L1/L2) allowing agents to retrieve relevant historical data efficiently.
*   **Context Processing:** Transforming raw retrieved data into actionable context for downstream agent consumption.
*   **Infrastructure Orchestration:** Managing the lifecycle of required backend binaries (Qdrant, Cortex-Mem) transparently.
*   **Legacy Migration:** Facilitating the transition of data from previous OpenClaw versions to the new tenant-isolated structure.

### 1.3 Technical Characteristics Overview
*   **Architecture Style:** Modular Plugin Architecture with Dual-Entry Points (Plugin & Context-Engine).
*   **Primary Language:** TypeScript (ensuring type safety and maintainability).
*   **Deployment Model:** Hybrid (TypeScript logic managing native binary processes).
*   **Communication Patterns:** Asynchronous for service lifecycle; Synchronous for critical configuration parsing.

## 2. Target Users

Based on the system's operational flows and configuration requirements, the target users and actors are categorized as follows:

### 2.1 Primary Actors: AI Agents
*   **Role:** Consumers of context and memory services.
*   **Usage Scenarios:** 
    *   Initiating semantic search queries to recall past interactions.
    *   Committing session data to memory after task completion.
    *   Retrieving contextual information to inform decision-making.
*   **Requirements:** Low-latency access to vector indices, reliable API contracts, and consistent data formatting.

### 2.2 Secondary Actors: Platform Administrators & Developers
*   **Role:** Configurers and maintainers of the MemClaw instance.
*   **Usage Scenarios:**
    *   Defining configuration schemas via TOML files.
    *   Triggering legacy data migration processes.
    *   Monitoring health checks of orchestrated backend services.
*   **Requirements:** Validatable configuration paths, clear error reporting during initialization, and idempotent migration tools.

## 3. System Boundaries

### 3.1 System Scope Definition
MemClaw is defined as the set of components responsible for memory storage orchestration, context retrieval logic, and configuration management. It sits between the AI Agent/Host Environment and the underlying data infrastructure.

### 3.2 Included Core Components (Inside Boundary)
*   **Configuration Management:** Handles TOML validation, path resolution, and settings synchronization.
*   **System Orchestration:** Manages binary discovery, process spawning, and health monitoring of backend services.
*   **Core Context Engine:** Implements semantic search logic, HTTP client facades, and tool registration for agents.
*   **Migration & Compliance:** Handles data transformation from legacy formats and enforces guideline injection.

### 3.3 Excluded External Dependencies (Outside Boundary)
*   **AI Models (LLMs):** MemClaw provides context to models but does not host or execute inference.
*   **Operating System:** MemClaw runs on top of an OS but does not manage OS-level resources beyond process spawning.
*   **User Interfaces:** MemClaw exposes APIs/plugins; it does not include a direct GUI for end-users.
*   **Network Infrastructure:** While it uses HTTP, the underlying network routing is handled externally.

## 4. External System Interactions

MemClaw interacts with several external software systems and data stores to fulfill its business functions. These interactions define the system's dependency graph.

### 4.1 External System List

| System Name | Type | Interaction Method | Purpose |
| :--- | :--- | :--- | :--- |
| **OpenClaw Host** | Software System | Plugin API / Hooks | Provides the execution environment; receives registered tools and context. |
| **Qdrant** | Vector Database | Local Process / Network | Stores vector embeddings for semantic search (Tiered Indexing). |
| **Cortex-Mem Service** | Microservice | REST API (HTTP) | Handles memory logic, session commits, and context processing. |
| **Legacy Workspace** | Data Store | File System Access | Source directory for migrating logs and preferences during upgrade. |

### 4.2 Dependency Relationship Analysis

1.  **MemClaw ↔ OpenClaw Host:**
    *   **Dependency:** High. MemClaw loads as a plugin.
    *   **Flow:** Host initializes MemClaw; MemClaw registers tools (e.g., `semanticSearch`) back to the host.
2.  **MemClaw ↔ Cortex-Mem Service:**
    *   **Dependency:** Critical. Core Context Engine relies on this service for all memory operations.
    *   **Flow:** MemClaw constructs typed HTTP requests (`fetchJson`, `sessionCommit`) to the service.
3.  **MemClaw ↔ Qdrant:**
    *   **Dependency:** High. Used for underlying vector storage.
    *   **Flow:** Orchestrated by System Orchestration module; queried indirectly via Cortex-Mem or directly for index regeneration.
4.  **MemClaw ↔ Legacy Workspace:**
    *   **Dependency:** Transient. Only active during migration workflows.
    *   **Flow:** Migration & Compliance module reads legacy paths and writes to tenant-isolated directories.

### 5.1 Key Interaction Flows
1.  **Initialization Flow:** 
    *   OpenClaw Host triggers `onLoad`.
    *   MemClaw validates TOML config (Configuration Management).
    *   MemClaw spawns Qdrant/Cortex binaries (System Orchestration).
    *   MemClaw exposes tools to OpenClaw Host.
2.  **Retrieval Flow:**
    *   Agent requests context via OpenClaw.
    *   MemClaw routes request to Cortex-Mem Service.
    *   Cortex-Mem queries Qdrant for vectors.
    *   Results processed and returned to Agent.
3.  **Migration Flow:**
    *   Admin triggers migration.
    *   MemClaw locates legacy paths via Config Resolver.
    *   Data Migrator copies logs/preferences to isolated tenant dirs.
    *   CLI Wrapper regenerates indices in Qdrant.

## 6. Technical Architecture Overview

### 6.1 Main Technology Stack
*   **Runtime:** Node.js (implied by TypeScript/binaries management).
*   **Language:** TypeScript (Primary logic, type safety).
*   **Configuration:** TOML (Schema validation, platform-specific path resolution).
*   **Protocols:** HTTP/REST (Client-Server communication), IPC (Binary spawning).
*   **Database:** Qdrant (Vector Storage), SQLite/File System (Local configs/logs).

### 6.2 Architecture Patterns
*   **Plugin Architecture:** MemClaw is designed to be loaded dynamically into the OpenClaw host, ensuring loose coupling.
*   **Dual-Entry Point:** Utilizes both `plugin/index.ts` (for host integration) and `context-engine/index.ts` (for core logic isolation).
*   **Facade Pattern:** The `HTTP Client Facade` abstracts complex REST interactions into typed functions (`fetchJson`, `semanticSearch`).
*   **Orchestration Pattern:** The `Binary Manager` handles the lifecycle of external processes, ensuring availability before accepting traffic.

### 6.3 Key Design Decisions
1.  **Separation of Concerns:** Distinct domains (Configuration, Orchestration, Core Engine, Migration) prevent monolithic complexity and allow independent scaling or refactoring.
2.  **Native Binary Management:** Rather than relying on containerization (Docker) for Qdrant/Cortex, MemClaw manages native binaries directly. This reduces overhead but increases platform dependency handling (handled via `bin-darwin-arm64/package.json`).
3.  **Tenant Isolation:** Migration logic enforces strict directory separation for different tenants/workspaces to ensure data privacy and integrity during the transition from legacy systems.
4.  **Hybrid Execution Model:** 
    *   *Synchronous:* Used for configuration parsing to fail fast on startup errors.
    *   *Asynchronous:* Used for service lifecycle and HTTP calls to maintain non-blocking performance.

### 6.4 Risks and Considerations
*   **Binary Compatibility:** Managing native binaries across platforms (e.g., Darwin ARM64) requires rigorous version control and compatibility testing.
*   **Service Health:** The Core Context Engine depends entirely on the health of Cortex-Mem and Qdrant. Robust retry logic and circuit breakers are implied in the `HTTP Client Facade`.
*   **Migration Complexity:** Regex-heavy components in the migration module present a refactoring opportunity to reduce maintenance debt.
